ARCHITECTURE PRINCIPLES:
- Keep the backend simple and consolidated — prefer fewer files over many small ones. One route file with all endpoints is better than one file per resource, unless the project genuinely has 10+ resources.
- SQLite does not support JSON column types. Use Text columns with JSON-serialized strings for structured data.
- NEVER use browser-side SQLite (sql.js, absurd-sql, wa-sqlite, or any WebAssembly SQLite). When the requirements say "SQLite," they mean a server-side SQLite file accessed by Flask via SQLAlchemy. ALL persistent data must live on the Flask backend — the frontend fetches it via the REST API. The frontend has NO direct database access of any kind.
- The import graph is critical — developers will copy your imports verbatim. Be explicit about every import path, including relative vs. absolute.
- Every entry point (run.py, main.dart, app.py, index.js) must work when executed directly — include any sys.path setup, environment loading, or bootstrap needed.
- For React projects: the React entry point is ALWAYS `frontend/src/App.jsx`. The scaffold provides `frontend/src/main.jsx` which imports and mounts `App.jsx` — do NOT replace or rename this contract. The react_dev must write `App.jsx` as the root component (with all routing, providers, and layout). Do NOT use `index.jsx` or any other file as the root. `main.jsx` already exists and imports `./App` — writing anything else as the entry breaks the app silently.
- For any algorithm, calculation, or non-trivial logic: specify the EXACT method, not just the interface. Two developers given the same function signature will implement different algorithms — and different algorithms produce different outputs, breaking tests.
- Design for ONE-SHOT implementation. Each developer agent writes their files once, completely, with no follow-up. Do not design architectures that require scaffold-then-fill patterns. Every file should be writable from scratch in a single pass with no dependency on a previous stub. Keep the file count low — 3–8 files per agent is ideal. More files = more things that can go wrong.
